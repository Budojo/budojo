## What

Apply Laravel's standard `throttle:5,1` middleware to `POST /api/v1/auth/login`, capping it at 5 attempts per minute per IP. Adds a 3-test PEST feature spec covering the 429 response, the credential-agnostic counting behaviour, and the happy-path sanity check. Updates the OpenAPI spec to surface the new 429 response.

## Why

Without a limiter the login endpoint is brute-forceable at network speed against any known email — the controller returns 401 in O(ms) and there is no inherent backoff. `/auth/forgot-password`, `/feedback`, and `/me/deletion-request` are all throttled today; login was the outlier. The cap is loose enough that an honest user fat-fingering their password 3–4 times still gets through, but a script grinding passwords hits 429 within seconds.

## How

- `server/routes/api_v1.php` — chained `->middleware('throttle:5,1')` on the existing login route. No named limiter; Laravel's default `throttle:N,M` is exactly the right shape for this.
- `server/tests/Feature/Auth/LoginThrottleTest.php` (new) — three feature tests using `RefreshDatabase`. `beforeEach` AND `afterEach` both clear the `throttle:5,1` limiter so this file (which deliberately exhausts the cap) doesn't leave state behind for the next file. `LoginTest::beforeEach` also clears the same key for the same reason — the suite is now order-independent.
- `docs/api/v1.yaml` — added a 429 response on `POST /auth/login` with the standard throttle envelope: `application/json` body referencing `ErrorMessage` (example: `{ "message": "Too Many Attempts." }`) and the `Retry-After` header documented as an integer.

### Throttle key choice — IP, not email

Keyed on IP (Laravel's default for unnamed throttles) on purpose. Email-keyed throttling would let an attacker lock out a known account by spamming its email from a botnet — a denial-of-service vector that's worse than the brute-force problem we're trying to solve. IP-keying matches what `/auth/forgot-password` does and what most off-the-shelf auth stacks default to.

## Notes

- Successful logins count toward the cap. This is Laravel's default — the limiter is keyed on the request signature, not the response status. A normal session never approaches 5 logins/minute, so the user-visible cost is zero.
- The `Retry-After` header on 429 is automatic from the middleware — clients (incl. the SPA) can render a countdown without extra wiring.
