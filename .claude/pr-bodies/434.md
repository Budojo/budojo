## What

Adds the two missing public-error landing pages the SPA was lacking:

- **`/error`** ‚Äî `ServerErrorComponent`, **reachable by direct navigation only** (typed URL, future deep-links from empty states / tooltips).
- **`/offline`** ‚Äî `OfflineComponent`, reached automatically when an outgoing API request fails with `HttpErrorResponse.status === 0` (no network).

Both mirror `NotFoundComponent` (#226) in shape: brand glyph + sentence-case title + supporting copy + a small set of CTAs. EN + IT i18n keys land in lock-step. The 404 page already existed; this PR fills the rest of the matrix.

A functional `errorInterceptor` (`HttpInterceptorFn`) handles the **network-unreachable** case (`status === 0`) and re-throws the original error so per-feature handlers still see it. `OnlineStatusService` exposes `navigator.onLine` as an Angular signal so the offline page auto-recovers the moment the browser fires the `online` event. The interceptor uses `skipLocationChange: true` so the URL bar stays on the originally failing route ‚Äî `location.reload()` and the auto-recovery effect therefore re-fire **the request the user actually wanted**, not the offline page itself.

## Why

#425 punch list: the SPA only had a 404 landing ‚Äî there was no UX for "you have no network" and no destination to deep-link from a hard-failure state. Without these the user experiences a half-broken dashboard with stale state, no recovery affordance, and no signal that the failure is global rather than a single-screen bug.

## How

### What auto-redirects, and what does NOT

- **`status === 0`** ‚Üí `/offline` (auto). Browser-emitted, request never reached the server (CORS, DNS, network drop, refused). This is a global condition ‚Äî the *whole* app is offline, not just one call ‚Äî so a full-page takeover is the correct UX.
- **5xx (500‚Äì599)** ‚Äî **NO auto-redirect.** Server-side errors are usually scoped to the endpoint that failed, and the codebase already wires component-level handlers (toasts on save failures, empty-state cards on list failures, the feedback page's "shows the error toast and keeps form contents on a server fail" assertion, etc.). A global redirect would silently break all of those by replacing the rendered component before the local handler fires. Components keep ownership of 5xx; `/error` stays routable for direct navigation but is no longer the destination of a global redirect.
- **4xx** propagates untouched. Login, form validation, 403 `verify_required` in the auth interceptor ‚Äî all of those own their own UX.

This is a deliberate narrowing from the first cut of the PR (which globally redirected on every 5xx). Copilot's review caught the design tension on the first pass; the regression on `feedback.cy.ts` confirmed the cost.

### `skipLocationChange: true` is load-bearing

Without it the browser URL bar would flip to `/offline`, so the retry button's `document.location.reload()` (and the auto-recovery effect on reconnect) would only ever reload the offline page itself ‚Äî never the URL the user was actually trying to reach. With the flag the URL bar stays on the originally requested route while Angular renders `<app-offline>`; reload returns the user to where they wanted to go. `Router.url` still resolves to the rendered route, so the loop guard inside the interceptor keeps working unchanged.

### Loop guard

The interceptor short-circuits the redirect when the user is already on `/offline` ‚Äî without that gate a status-0 retry on `/offline` would loop. The `OfflineComponent`'s auto-recover effect skips the initial run when the browser is already online (a stale redirect that lands the user there optimistically), so a user who hits "Try again" doesn't get bounced again on the next render.

### Cypress harness ‚Äî default `/me` mock

The dashboard shell calls `loadCurrentUser()` (`GET /api/v1/auth/me`) on init. Most existing E2E specs don't mock `/me` (they didn't have to before this PR; the dev server proxy used to silently swallow the failure). With the new `errorInterceptor`, an unmocked `/me` in CI would render `<app-offline>` in place of the page under test. To prevent that without churning every spec, `cypress/support/e2e.ts` registers a default `cy.intercept('GET', '/api/v1/auth/me*', ‚Ä¶)` in a global `beforeEach`. Specs that need a custom `/me` (Mario Rossi for `profile-mobile`, the unverified-user case for the verification banner) register their own intercept later ‚Äî Cypress matches the most-recent registration first.

### Tests

- Vitest: 5 cases on the interceptor (every 5xx ‚Üí NOT redirected, every 4xx ‚Üí NOT redirected, status 0 ‚Üí /offline, no loop on /offline, error re-thrown).
- Vitest: 5 cases on `OnlineStatusService` (initial state from `navigator.onLine`, signal flip on browser events, SSR-safe `defaultView = null` path).
- Vitest: 4 cases per error component (renders, CTAs present, retry reloads, navigate-home / auto-recovery).
- Cypress: visit `/error` and `/offline`, assert content, assert "Back to home" navigates away.

## Notes

- All visible strings live in `errorPage.*` and `offlinePage.*` i18n keys (EN + IT in lock-step). Copy is honest about what the implementation actually does ‚Äî no "team has been notified" claims (no error reporting hook is wired in this PR).
- The `BrandGlyphComponent` import + visual layout matches `NotFoundComponent` exactly ‚Äî no new design tokens, no new partial styles.
- The offline page only exposes "Try again" ‚Äî there is no "Back to home" because every navigation would fail without the network. This is intentional (Don't Make Me Think ¬ß self-evidence).
- The interceptor is registered AFTER `authInterceptor` in the `withInterceptors([...])` array so it sees post-auth-mutation responses.
