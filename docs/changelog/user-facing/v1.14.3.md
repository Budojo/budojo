# v1.14.3 — 2026-05-04

The actual fix for the **Stats** page blank-on-first-click that v1.14.1's preload change tried — and failed — to nail. Clicking "Stats" in the sidebar after navigating around the dashboard now lands on the page first time, every time, with no detour through F5.

## 🐛 Stats blank page on first in-app navigation — fixed

- **Direct cause: a defensive `?` missing in one place.** The Stats parent page reads the active tab from the current URL the moment it mounts. Under certain timings — specifically when entering Stats from another dashboard page, with the "preload everything" behavior introduced in v1.14.1 — the route information the page reads from is briefly in a half-built state. The previous code assumed it was always fully populated and crashed silently on the missing field, leaving the dashboard chrome on screen and the content area blank.
- **Three more `?` characters and the chain falls back gracefully.** With the fix, every step of the lookup is now optional, so any transient half-state cleanly falls back to the default "Overview" tab and the page renders normally on first try. Hard refresh (F5) is no longer required.
- **Regression-pinned.** A new test simulates the exact half-built route state that crashed prod and asserts the page still renders cleanly. So if a future change re-introduces the same shape of bug, CI catches it before it reaches you.

## 🧹 Behind the scenes (continuing v1.14.2's work)

- v1.14.2 (released earlier the same day) shipped an auto-recovery safety net for stale-bundle navigation failures — a browser tab whose cached bundle stopped matching the deployed code now reloads itself once instead of staying blank. That code is unrelated to this fix and stays as belt-and-braces for a different class of cache-related failure.
