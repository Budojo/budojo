# Design inventory — screenshot library

This folder holds screenshots of every canonical page of the Budojo SPA
at three viewport widths (mobile / tablet / desktop), regenerated by a
single command. It exists so we always have an up-to-date visual
reference to compare new features against — "what does a list page
look like in Budojo?" is answered by scrolling this folder, not by
guessing.

## When to regenerate

Run `npm run design:inventory` (from `client/`) after any of:

- A new page / feature screen is added (see the playbook below).
- Global styles change (design tokens, theme, typography).
- A shared component's visual footprint changes (belt badges, tags,
  cards).
- Before opening a PR that visibly touches the UI — commit the updated
  screenshots alongside the code change so reviewers can see the diff.

## How to regenerate

Prerequisites: the dev server is up and serving on `localhost:4200`
(usually via `docker-compose up budojo_client`).

```bash
cd client
npm run design:inventory
```

The script runs `cypress run` against a single spec —
`cypress/e2e/design-inventory.cy.ts` — which visits every declared page
at three viewports and writes `{slug}__{viewport}.png` here.

The inventory spec is excluded from the default Cypress test glob (see
`cypress.config.ts` `excludeSpecPattern`), so running `npm run cy:run`
or CI never touches these files. Regeneration is deliberate, on-demand.

## Naming convention

Files follow `{route-slug}__{viewport}.png`:

- `{route-slug}` — the route path with slashes replaced by dashes, e.g.
  `/dashboard/academy` → `dashboard-academy`, `/dashboard/academy/edit`
  → `dashboard-academy-edit`.
- `{viewport}` — one of `mobile` (390 × 844), `tablet` (768 × 1024),
  `desktop` (1280 × 800).

Example: `dashboard-athletes-list__mobile.png`.

Sorting alphabetically groups all viewports of a single page together,
which is exactly the comparison you usually want.

## Playbook — adding a new feature

When you're about to build a new screen (e.g. M4 Attendance, future
Classes / Lessons, a Reports dashboard):

1. **Read the canon first.** Open
   [`../DESIGN_SYSTEM.md`](../DESIGN_SYSTEM.md) and scan for the closest
   existing pattern. Grouped-list card? Filled alert widget? Bordered
   table? Reuse the pattern before inventing one.
2. **Scan this folder.** Find a screen with a similar shape and borrow
   its structure — headers, spacing, action placement. Consistency
   across the app is worth more than novelty in any one place.
3. **Build the component** using `var(--p-*)` and `var(--budojo-*)`
   tokens only. If you catch yourself writing a raw hex color or an
   invented spacing value (e.g. `0.75rem`), stop — the token exists,
   find it.
4. **Add the page to the inventory spec** — open
   `client/cypress/e2e/design-inventory.cy.ts` and add one line:

   ```ts
   captureAtAllViewports('/dashboard/lessons', 'dashboard-lessons-list');
   ```

   Add deterministic API intercepts if the page fetches anything not
   already stubbed in `seedIntercepts()`.
5. **Regenerate**: `npm run design:inventory`. Fresh PNGs appear here.
6. **Review against the existing inventory.** Open the new screenshot
   side by side with the closest existing page. Ask:
   - Are the radii, spacing, elevations, typography *identical*?
   - Does the primary CTA match the other primary CTAs in the app?
   - Do the tags match the tag canon?

   If your new screen needs a pattern the system genuinely doesn't
   have (e.g. a calendar cell, a chart legend), don't invent one in
   this PR. Flag the gap, go back to Claude Design with a specific
   brief ("I need a calendar cell variant for attendance check-in,
   the closest existing pattern is the grouped-list cell, but…"),
   and extend the design system deliberately.
7. **Commit the updated screenshots** in the same PR that ships the
   component code. The reviewer gets to see the visual delta, not
   just the HTML/SCSS.

## What's NOT covered yet

- **Auth / setup pages** (`/auth/login`, `/auth/register`, `/setup`)
  use a pre-design-system purple/pink branding that predates the
  tokens. Intentionally excluded from the inventory until they're
  refactored onto the design system — otherwise their inclusion would
  mislead "what does Budojo look like" by showing two incompatible
  visual directions.
- **Dark mode.** The design system defines a dark mode (`.dark` class
  on `<html>`) but the app has no toggle yet. When it ships, the
  inventory spec should gain a second pass with `cy.visit(url, {
  onBeforeLoad(win) { win.document.documentElement.classList.add('dark'); } })`.
- **Dialogs and popovers** are tricky to screenshot because they're
  triggered by interaction. A future pass could add "scenario"
  screenshots (e.g. athlete delete confirm open) via explicit
  `cy.click()` + wait before screenshotting.

## Size discipline

Committed PNGs at ~3 viewports × ~10 pages = ~30 files; at ~200 KB
each that's ~6 MB in git. Acceptable for a reviewed visual library.
Do NOT commit screenshots for every intermediate WIP state —
regenerate only when things change meaningfully and the PR will land.
